# 智能哨兵 系统设计文档


## 一、应用信息

- 应用名: AlarmGuard

- 模块名: WorkOrderMgmt

- 两个页面：报警记录、报警工单

- Node-RED pgconfig ID: pgconfig_m8b4w

- 流文件: flows/AlarmGuard_WorkOrderMgmt.json

- 技术实现概要：从外部报警历史接口分页获取报警记录，选中该记录后，点击生成报警工单（该报警记录一旦生成工单，则该报警记录不要再出现“报警记录页面”里），进行工单存储，并且能进行工单修改、删除，而且当工单处理状态设置为已确认时，调用下写接口（UNS External Action，通过 Node-RED 代理转发）关闭声光报警。

## 二、菜单结构
1. 智能哨兵-报警记录
2. 智能哨兵-报警工单


## 三、页面

### 1、报警记录页面

#### 页面功能

1. 初始加载：使用 `useEffect(() => { loadAlarms(1); }, [])` 触发（⚠️ 不能用 `useState` 回调，会导致无限渲染）

2. 调用代理端点 `POST /AlarmGuard/WorkOrderMgmt/Action/alarm_history_proxy` 获取报警历史

3. 服务端过滤+服务端分页（每页 20 条）：位号（sourceNameLike）、报警描述（keyword 匹配 description）、产生时间区间（activeTimeBegin / activeTimeEnd），所有过滤参数随分页请求一起发送至代理端点，由外部接口执行过滤

#### 检索条件

1. 位号（模糊匹配） 

2. 报警描述（模糊匹配）

3. 报警产生时间（区间筛选）


#### 表格列

序号、位号（sourceName）、详细描述（description）、报警子类型（detailSubType）、优先级（priority）、报警值（value.value）、触发条件（definition?.limit ?? '-'）、路径（sourcePath）、报警时间（activeTime）

#### 操作按钮

1. 详情：用于查看报警详情（按照报警返回结果查看报警详细信息页：报警位号、源路径、具体类型、报警描述、一级分类、子类型、逻辑分组、优先级、报警触发（触发属性、报警值、数字质量）、报警配置（conditionDefine）、是否已恢复）
2. 生成工单：前端生成工单编码（`AROrder + YYYYMMDD + 4位随机数字`），调用 POST 新建工单接口写入数据库，跳转至工单页

### 2、报警工单页面（AlarmWorkOrders.tsx）

#### 检索条件

1. 工单编码（模糊匹配）

2. 报警位号（sourceName 模糊匹配 ）
3. 工单处理状态（下拉：全部 / 未处理 / 已确认）

#### 表格列

序号、工单编码、报警位号、报警描述、工单处理状态、问题原因、纠正预防措施、确认时间、附件（点击下载）、备注

#### CRUD 功能

##### 修改（弹窗）

修改字段：工单处理状态、问题原因、纠正预防措施、备注、附件

##### 删除（逻辑删除）

规则：delete_flag=1 + delete_time=NOW()，所有 GET 查询过滤 WHERE delete_flag=0

##### 导入（CSV）

1. 规则：按工单编码匹配更新，不存在则跳过并提示错误

2. 提供模板下载功能

##### 导出（CSV）

导出当前列表数据

#### 附件处理规范

1. 读取方式：FileReader.readAsDataURL() → Base64 格式

2. 存储格式：JSON 数组字符串存入 attachments TEXT 字段

```json
[{"name":"文件名","size":字节数,"type":"MIME类型","dataUrl":"data:...;base64,..."}]
```
3. 单文件大小限制：5MB


#### 工单详情页（WorkOrderDetail.tsx）

##### 页面结构

1. 顶部：工单基本信息卡片（含附件列表，支持点击下载）

2. 下方页签：

    - 历史报警：调用代理端点，按 source_Path 过滤该位号的所有报警历史

    - 历史工单：查询工单表，按 source_path 过滤该位号的所有工单记录


## 四、技术要求

### 1、报警历史接口

1. 分页调用接口示例：

```bash
curl --location --request post 'https://bootcamp.devcloud.supos.net/os/open-api/alarm/history/v1/raw' \
--header 'Authorization: Bearer ak-7442770144515851191'
```

2. 请求参数：

```json
{
  "clientId": "production-monitor-001",
  "pageNum": 1,
  "pageSize": 2,
  "sortBy": "activeTime",
  "sortOrder": "desc",
  "inactived": true
 
}

```
	
过滤查询示例：
 "activeTimeBegin":"2024-08-31T00:00:00.000+08:00", //报警激活时间下限（>= 该时间）
 "activeTimeEnd":"2024-09-01T00:00:00.000+08:00", 报警激活时间上限（<= 该时间）
  "sourceNameLike":"T-101",  按报警源显示名称模糊查询
  "keyword":"温度过高" //关键字，用于模糊匹配报警源名称、路径、描述字段


3. 返回报警记录数据结构请调用接口分析其结构


### 2、下写数据接口（External Action 代理）

取消报警（工单确认时调用），通过 UNS External Action `cancel_alarm` 代理转发：

- 外部实际接口：`POST https://bootcamp.devcloud.supos.net/os/static/nodered/nodered/api/alarm/control`
- 前端调用路径：`POST /AlarmGuard/WorkOrderMgmt/Action/cancel_alarm`
- 请求体：`{"status": 0}`

UNS.json 中需定义 External Action topic：
```json
{
  "namespace": "AlarmGuard/WorkOrderMgmt/cancel_alarm",
  "name": "cancel_alarm",
  "displayName": "取消声光报警",
  "labels": "Action",
  "dataType": "2",
  "persistence": "FALSE",
  "source": "external",
  "apiConfig": {
    "url": "https://bootcamp.devcloud.supos.net/os/static/nodered/nodered/api/alarm/control",
    "method": "POST"
  },
  "definition": "[{\"name\":\"alarm_status\",\"type\":\"INTEGER\"}]",
  "type": "file"
}
```
> ✅ 下写接口统一通过 Node-RED 代理转发，避免前端硬编码外部 URL，便于环境切换和统一管控。



### 3、 ⚠️ 跨域（CORS）问题及解决方案

- 问题：浏览器直接调用**报警历史接口**会报 Network Error（CORS 限制），即便 Postman/curl 正常。

- 解决方案：通过 Node-RED 做服务端代理，前端调用本地代理端点：

1. 前端 → POST /AlarmGuard/WorkOrderMgmt/Action/alarm_history_proxy

2. → Node-RED function（组装 url/method/headers/payload）

3. → Node-RED http request 节点（method: "use", url: ""）

4. → Node-RED function（解包 body.data.records → { success, data, total, page, size }）

5. → 返回前端

> ✅ **下写控制接口（alarm/control）**同样通过 Node-RED 代理转发（External Action `cancel_alarm`），与报警历史代理方式一致。

### 4、 Node-RED http request 节点关键配置

|配置项|取值|说明|
|---|---|---|
|method|"use"|从 msg.method 读取|
|url|""|从 msg.url 读取|
|ret|"obj"|自动解析 JSON|
|senderr|true|错误时继续流程|
|timeout|30000|超时时间30秒|

### 5、 已知问题与修复记录

|问题|根因|修复方式|
|---|---|---|
|外部报警 API CORS 报错|浏览器直接跨域请求被拦截|添加 Node-RED 代理端点 alarm_history_proxy，前端改为调用本地路径|
|DataTable 192 条重复 key 警告|
|api.fetch 获取不到 total|api.fetch 自动解包 data 内层，导致 res.total === undefined|改用 api.fetchWithMeta|
|验证器报 alarm_history_proxy 404|UNS.json 中未注册该 Action topic|在 UNS.json 中添加 alarm_history_proxy Action topic|
|验证器报 GET /files 404|缺少文件服务 flow|从模板创建 flows/_fileService.json，替换 pgconfig ID|
|报警记录页面初始不加载|用 useState 回调触发异步请求导致无限渲染|改为 useEffect(() => { loadAlarms(1); }, []) 触发初始加载|
|下写接口前端硬编码外部 URL|前端直接 fetch 外部 URL，脱离 UNS 管控，环境绑定|改为 UNS External Action `cancel_alarm`，通过 Node-RED 代理转发，前端调用标准 Action URL|
